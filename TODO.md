## 当前待改进点

### 1. 进一步澄清“稳定配置直跑”的适用前提
- 在不增加太多 token 成本的前提下，进一步说明：
  - 首次接入用户环境时，仍需完成 Notion 凭证、数据源 ID、字段映射、时区确认
  - 只有在这些配置已经确认并稳定后，才适合跳过重复校验、直接执行
- 目标：避免公开仓库使用者误解为“任何环境都默认跳过确认”

### 2. 明确运行时到底读取源码目录还是打包产物
- 进一步梳理 OpenClaw / 本地运行环境对 skill 的加载路径：
  - 源码目录
  - `skills/` 入口
  - `.skill` 打包产物
- 目标：减少“改了仓库但运行时没生效”的歧义

### 3. 评估是否提交 dist 构建产物
- 明确仓库约定：
  - 是否提交 `dist/notion-smart-collector.skill`
  - 若提交，何时更新
  - 若不提交，如何在发布时生成
- 目标：降低 review 噪音，保持提交边界清晰

### 4. 继续压缩高频收藏场景的 token 消耗
- 对高频场景继续优化：
  - X 链接尽量只命中 1 次轻量接口
  - GitHub 链接尽量避免无谓正文抓取
  - 已知站点优先模板化生成一句简介
- 目标：把单条收藏稳定压到更低 token 区间

### 5. 为失败场景补更明确的降级策略
- 当前已有“写入失败再校验 schema”的方向，后续可继续细化：
  - 抓取失败时的回退链路
  - Notion 返回 4xx / 429 / 5xx 时的处理方式
  - 简介生成信息不足时的最小可用写入策略
- 目标：提升稳定性和可恢复性

### 6. 后续考虑加入更明确的本地缓存设计
- 目前“稳定配置复用”仍偏策略描述，后续可以继续细化：
  - 缓存哪些信息（字段映射 / 数据源 ID / 站点规则）
  - 缓存失效条件是什么
  - 失败后如何自动触发重校验
- 目标：把“省 token”从经验规则推进到更清晰的机制
